# Методика замера производительности TPU (cv181x / SG2002)

Продуманная воспроизводимая методика бенчмарка NPU и набор инструментов.
Цель это честно измерить, что TPU реально отдаёт на железе, отделить это от
маркетинговых TOPS, и сделать повторный замер дешёвым (одна команда). Все
инструменты в `tools/cviruntime/`. Durable-выводы в памяти
`tpu-real-max-int8-io-and-clock`. Замеры ниже это железо 2026-06-03.

## Принцип: пик против эффективного

TOPS бывает двух видов, путать нельзя.

- Паспортный пик (nameplate) это `2 × MAC_на_такт × частота`. Теоретический
  потолок при 100% загрузке MAC-массива. Не достигается ни одной реальной
  нагрузкой. Это «наклейка», не замер. У cv181x ~512 MAC/такт это 0.5 TOPS @500
  МГц (спека Sophgo), ~0.7 TOPS @700 МГц (наш FSBL perf).
- Эффективный (achieved) это `FLOPs_модели / время_TPU_forward`. Реальные
  полезные операции в секунду. Всегда заметно ниже пика. Это и есть, что меряем.

`FLOPs` берём из компилятора (tpu-mlir пишет `module.FLOPs` в `.mlir`,
конвенция MAC=2 op), НЕ из литературы. Знаменатель это изолированная фаза
TPU-forward (без pre/post), p50 из `tpu_bench`.

## Метрики (tpu_bench.cpp)

- латентность forward p50/p90/p95/p99/min/max/mean/stddev по N прогонам, warmup
  отдельно
- throughput fps (forward-only и pipeline)
- разбивка фаз CPU-pre (заливка входа) / TPU-fwd / CPU-post (постпроцесс)
- effective/nameplate/efficiency при `--flops F --clock-mhz M`
- сверка argmax между итерациями (дешёвый сигнал корректности под нагрузкой)

## Saturating-модели (почему именно они)

Реальная сеть типа mobilenet даёт лишь ~4% пика (depthwise, низкая
арифметическая интенсивность, упор в память). Чтобы увидеть достижимый максимум,
нужен compute-bound зонд с высокой интенсивностью (MAC на байт), держащий
MAC-массив занятым. Свип в `saturating/` (Conv1x1, Conv3x3, GEMM разных
размеров). Conv3x3 с большими каналами это самая высокая интенсивность это
ближе всего к пику. Это roofline-подход: меняем интенсивность, смотрим, где
упираемся.

## Ключевой урок: INT8 I/O обязателен

cvimodel по умолчанию имеет FP32 вход/выход (внутри INT8). Тогда на каждый
forward рантайм конвертит fp32<->int8 на CPU и синкает вчетверо большие буферы.
Это убивает throughput: forward становится data-bound (время = (вход+выход) /
скорость), вычисление TPU невидимо.

- FP32 I/O: conv1x1_c512 forward 681 мс, максимум по свипу 0.02 TOPS (3% пика)
- INT8 I/O (`model_deploy --quant_input --quant_output`): тот же conv 7.2 мс
  (95x быстрее), максимум 0.34 TOPS на conv3x3 (48% пика)

Любую модель для throughput деплоить с INT8 I/O. Перед выводами о «медленном
TPU» проверить формат I/O (`out fmt=0` в tpu_bench это FP32).

## Клок и разгон (set_tpu_clk.py)

TPU=700 МГц это TPLL/2 (FSBL perf, напряжение 0.96В). Выше обратимо муксом
`clk_tpu` через `/dev/mem` (то же, что clk-framework, read-modify-write
`REG_DIV_CLK_TPU`): mipimpll/1=900. Честный 1 ГГц только через FSBL (нет PLL на
1000/2000), с поднятием напряжения.

Протокол разгона (`run_900.sh`), обязательно с гейтом корректности:
- снять argmax на штатном клоке как ДИНАМИЧЕСКИЙ эталон (не жёсткое число)
- разогнать, потребовать ТОТ ЖЕ argmax. Совпал это компьютинг стабилен. Не
  совпал это нестабильно, авто-откат
- всё обратимо (revert сохранённого регистра либо перезагрузка)

Результат 2026-06-03: 900 на 0.96В НЕстабилен (argmax 904->0, обнуление
выхода). 700 это потолок без поднятия напряжения.

## Воспроизводимость (одна команда)

Хост (регенерация всего кита из исходников):
```
podman build -t tpu-mlir:local tools/tpu-mlir      # один раз
tools/cviruntime/make_kit.sh                        # -> tpu-kit.tar.gz
```

Плата (предусловия: soph_tpu загружен, CMA dma-heap, FSBL perf 700 МГц):
```
tar xzf tpu-kit.tar.gz && cd tpu-kit
./tpu_maxtops.sh                 # максимум на 700 МГц
./run_900.sh                     # разгон 900 с гейтом корректности (+ откат)
```

## Результаты 2026-06-03 (INT8 I/O, 700 МГц)

| Модель | FLOPs | fwd p50 | effective | % пика |
|---|---|---|---|---|
| conv3x3_c384_s48 | 6.12G | 17.9 мс | 0.342 TOPS | 47.7% |
| conv3x3_c256_s64 | 4.83G | 14.2 мс | 0.341 TOPS | 47.6% |
| conv1x1_c512_s64 | 2.15G | 7.2 мс | 0.299 TOPS | 41.7% |
| conv1x1_c256_s64 | 0.54G | 2.3 мс | 0.234 TOPS | 32.6% |
| gemm_m256_k1024 | 0.54G | 4.4 мс | 0.122 TOPS | 17.1% |

Максимум 0.34 TOPS = 48% от паспортных 0.7 @700 МГц. Для одиночной свёртки на
малом NPU это здоровая цифра. Контекст: «0.5 TOPS» чип на 500 МГц отдал бы
эффективно ~0.24, то есть наши 0.34 это больше реальной пропускной способности.

## Открытые рычаги (что ещё не выжато)

- Драйверный пер-forward синк: `cvi_tpu_prepare_buffer` делает
  attach+map_attachment+vmap+begin_cpu_access с `DMA_BIDIRECTIONAL` на КАЖДЫЙ
  submit. Фикс это маппить один раз + правильное направление DMA. Вторичный
  рычаг после INT8 I/O, может подтянуть effective к 0.4-0.5 @700. См.
  `patches/cvitek-tpu-vendor/`.
- Клок: 700 это потолок без напряжения. 1 ГГц только через FSBL + напряжение +
  охлаждение.
